54 /*
55  * reqt, reqf, p-idle, ec-idle, oq-empty, p-blocked, ec-idle, oq-empty messages?
56  *
57  * MESSAGE EXCHANGE STATES
58  * - READ In the READ state the I/O Control is reading data bytes and GET messages from the interface and placing
59  * them in the Input Buffer. The Parser and Execution Control are active and the Output Queue is empty. A brq
60  * cannot occur while in the READ state.
61  * The MEC remains in the READ state when either dcas or pon occurs. The device performs the INITIALIZE
62  * action described in 6.3.5.2.
63  * The MEC remains in the READ state when bav, get, eom, or ib-blocked occurs.
64  * The MEC enters the QUERY state when query occurs.
65  *
66  * - QUERY In the QUERY state the Parser has recognised a valid query within a <PROGRAM MESSAGE>. The I/O
67  * Control Emulator continues reading data bytes and GET messages from the interface and placing them in the
68  * Input Buffer.
69  * The MEC remains in the QUERY state when bav, get, query, or ib-blocked occurs.
70  * The MEC enters the SEND state when brq occurs.
71  * The MEC enters the RESPONSE state when eom occurs and ib-empty is TRUE.
72  * The MEC enters the READ state when an eom occurs and ib-empty is FALSE. This condition indicates that
73  * the controller started sending a new message before reading all of the response. The device performs the
74  * INTERRUPTED action described in 6.3.5.3.
75  * The MEC enters the DEADLOCK state when ib-full, oq-full, p-blocked, ec-blocked, and rf-blocked are all
76  * TRUE. This condition indicates that the controller is sending more data to the device and expecting more
77  * data from the device than the device can buffer. The device performs the DEADLOCK action described in
78  * 6.3.5.5.
79  * The MEC enters the READ state when either dcas or pon occurs. The device performs the INITIALIZE
80  * action described in 6.3.5.2.
81  *
82  * - SEND In the SEND state the device starts to send the <RESPONSE MESSAGE> to the controller. The device
83  * continues to receive data bytes from the interface, and to parse and execute the Query Message stored in the
84  * Input Buffer. The Response Formatter prepares the <RESPONSE MESSAGE>, and places it in the Output
85  * Queue.
86  * A deadlock situation is possible in the SEND state. The controller or communication channel may buffer
87  * part of the response even through the application has not requested any data. This buffering is actually an
88  * extension of the Output Queue, but the device has no control over it. If this extra buffer should fill, so the
89  * device is prevented from sending more data while the controller is still sending the <PROGRAM
90  * MESSAGE>, the Input Buffer may also fill, thus deadlocking the message exchange.
91  * The device could possibly detect this situation, but it cannot clear the extra buffers, so it merely stays in the
92  * SEND state until the controller clears the deadlock.
93  * The MEC remains in the SEND state when bav, get, brq, query or ib-blocked occurs.
94  * The MEC remains in the SEND state when ib-empty becomes TRUE. This transition might be considered
95  * UNTERMINATED since it appears that the controller failed to send an eom before it started to read the
96  * response. The nature of a full-duplex serial interface, however, allows bytes to be transferred and buffered in
97  * both directions. The eom may still be on its way from the controller so the device waits for it in SEND.
98  * The MEC enters the RESPONSE state when eom occurs and ib-empty is TRUE.
99  * The MEC enters the READ state when eom occurs and ib-empty is FALSE. This condition indicates that the
100  * controller started sending a new message before reading all of the response. The device performs the
101  * INTERRUPTED action described in 6.3.5.3.
102  * The MEC enters the READ state when either dcas or pon occurs. The device performs the INITIALIZE
103  * action described in 6.3.5.2.
104  *
105  * - RESPONSE In the RESPONSE state the Parser has received a <PROGRAM MESSAGE TERMINATOR> and the device
106  * is sending the complete <RESPONSE MESSAGE> to the controller. The Response Formatter formats the
107  * <RESPONSE MESSAGE> and the I/O Control sends data bytes from the Output Queue to the controller.
108  * An eom, query, or ib-blocked cannot occur because the Input Buffer is empty.
109  * The MEC remains in the RESPONSE state when a brq occurs.
110  * The MEC enters the READ state when RMT-sent occurs.
111  * The MEC enters the READ state when bav or get occurs. This condition indicates that the controller started
112  * sending a new message before reading all of the response. The device performs the INTERRUPTED action
113  * described in 6.3.5.3 of this standard.
114  * The MEC enters the READ state when either dcas or pon occurs. The device performs the INITIALIZE
115  * action described in 6.3.5.2.
116  * A device may have nothing to say while the MEC is in the RESPONSE state if all of the queries in the Query
117  * Message failed to generate any <RESPONSE DATA> because of syntax (Command), semantic (Execution),
118  * or Device-specific errors. The device does not send a <RESPONSE MESSAGE TERMINATOR> to the
119  * controller. The MEC remains in the RESPONSE state until either bav, get, pon, or dcas occurs.
120  *
121  * - DEADLOCK In the DEADLOCK state the device has been asked to buffer more data than it has room to store. The Output
122  * Queue was full, blocking the Response Formatter, Execution Control, and Parser. The Input Buffer was full,
123  * and the controller was waiting to send more data bytes to the device.
124  * The MEC breaks the deadlock by clearing the Output Queue and resetting the Response Formatter. The
125  * MEC also reports a Query Error to the Status Reporting. While in the DEADLOCK state, the MEC continues
126  * to parse and execute <PROGRAM MESSAGE UNIT> elements as usual, except that it discards query
127  * responses rather than placing them into the Output Queue.
128  * Because the Output queue is kept empty in DEADLOCK, brq can never occur.
129  * The MEC remains in the DEADLOCK state when a bav, get, query or, ib-blocked occurs.
130  * The MEC enters the READ state when eom occurs.
131  * The MEC enters the READ state when either dcas or pon occurs. The device performs the INITIALIZE
132  * action, described in 6.3.5.2.
133  *
134  */
